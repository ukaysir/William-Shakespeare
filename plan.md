# 공모전 기획서 작성 Agent Skill 생성 계획

## 1. 프로젝트 목표

공고문, 평가기준, 근거자료, 아이디어 또는 초안을 입력받아 심사자가 제안을 빠르게 이해하고 평가항목별 근거를 쉽게 찾을 수 있도록 기획서를 설계·진단·재작성하는 Codex Skill을 만든다.

작업명은 우선 `write-competition-proposals`로 사용한다. 수상을 보장한다는 인상을 피하면서도, 한국어 공모전 기획서의 작성·진단·개선 작업에 정확히 반응하게 만드는 것이 목표다.

### 한 문장 임무

> 심사자가 30초 안에 문제–대상–해법–효과를 이해하고, 평가항목별 채점 근거를 바로 찾을 수 있는 공모전 기획서를 만든다.

### 성공 조건

- 제목과 첫 문단만으로 문제, 대상, 제안, 기대 변화가 식별된다.
- 공고문의 모든 평가항목에 대응하는 근거 또는 명시적인 정보 공백이 존재한다.
- 전략·근거 부족과 표현 문제를 구분한다.
- 없는 통계, 사례, 실행계획을 만들어내지 않는다.
- 문단마다 하나의 중심 질문과 답이 있고, 다음 문단으로 논리가 전진한다.
- 동일한 대상, 용어, 수치, 단위, 기간이 문서 전체에서 일관된다.
- 진단 결과마다 원문 위치, 판정 이유, 수정 방향이 제시된다.
- Skill 폴더가 공식 검증 스크립트를 통과한다.

## 2. V1 범위

### 지원할 작업

1. 공고문·평가표·제출양식 분석
2. 아이디어를 제안 구조와 목차로 변환
3. 기획서 초안 작성
4. 기존 초안 진단
5. 사실을 보존한 부분 재작성 또는 전체 재작성
6. 평가항목별 근거 배치 검사
7. 제출 전 논리·표현·근거·일관성 최종 검수

### 우선 대상

- 한국어 서면형 아이디어 공모전
- 정책·공공서비스·사회문제 해결형 공모전
- 서비스·사업·제품 기획형 공모전
- 평가항목과 배점이 제시되는 제안서형 공모전

### V1 비범위

- 소설, 수필, 시나리오 등 문예 공모전
- PPT 시각 디자인과 발표 코칭
- HWP·DOCX의 정교한 서식 자동 편집
- 시장조사·통계분석 자체를 전담하는 기능
- 법률·재무·기술 타당성에 대한 전문가 보증
- 수상 가능성 또는 심사 점수의 확정적 예측

필요성이 확인되면 후속 버전에서 발표자료, 문서 서식, 데이터 검증 기능을 별도 모듈로 확장한다.

## 3. 핵심 설계 원칙

### 3.1 평가기준 우선

일반적인 좋은 글보다 해당 공모전의 공식 평가항목과 배점을 우선한다. 평가항목을 Skill에 고정하지 않고 매 작업에서 공고문으로부터 추출한다.

### 3.2 이해는 설득의 관문

문장 미화보다 독자가 문제–대상–해법–작동원리–효과를 정확히 재구성할 수 있는지를 우선한다.

이해에 그치지 않고 심사자가 `왜 지금 이 문제를 해결해야 하며 왜 이 제안을 선택해야 하는가`를 원문 근거로 판단하게 한다. 제출 원고는 근거가 허용하는 가장 강한 주장을 사용하고, 한계·미확정·확인 사항은 심사 판단을 바꾸는 경우가 아니면 내부 감사 기록에만 보관한다.

### 3.3 네 종류의 결함 분리

- `CONTENT_GAP`: 계획, 근거, 차별성 또는 실행 정보 자체가 부족함
- `WRITING_GAP`: 필요한 정보는 있으나 구조나 표현 때문에 전달되지 않음
- `EVIDENCE_RISK`: 출처, 기준시점, 모집단, 인과관계 또는 수치가 불명확함
- `COMPLIANCE_RISK`: 자격, 분량, 서식, 필수항목 또는 제출조건 위반 가능성이 있음

`CONTENT_GAP`은 임의로 보완하지 않고 `[근거 필요]`, `[수치 확인 필요]`, `[실행주체 미정]`처럼 표시한다.

### 3.4 근거와 주장 강도 보존

사실, 해석, 추정, 목표, 가정을 구분한다. 근거가 목표만 뒷받침하는 경우 “달성한다”로 단정하지 않는다.

### 3.5 심사자 중심 출력

원고의 아름다움보다 심사자가 다음 행동을 쉽게 수행하도록 한다.

- 제안의 핵심을 설명하기
- 평가항목별 강점과 약점을 찾기
- 수치와 실행계획을 확인하기
- 점수를 부여할 근거를 인용하기

### 3.6 점수보다 증거

근거 없는 총점이나 `87점` 같은 정밀 점수를 피한다. 항목별로 `0=부재 또는 모순`, `1=부분 충족`, `2=명확히 충족`을 사용하고, 반드시 원문 근거를 붙인다.

우선순위는 별도로 표시한다.

- `P0`: 제출 자격, 사실성, 핵심 이해를 무너뜨리는 결함
- `P1`: 평가항목 점수에 직접 영향을 주는 결함
- `P2`: 가독성·완성도를 낮추는 결함

### 3.7 점진적 공개

`SKILL.md`에는 핵심 절차와 참조 경로만 둔다. 세부 판정 규칙, 예시, 출력 형식은 `references/`로 분리한다. `SKILL.md`는 500줄 미만을 유지한다.

## 4. 입력 계약

Skill은 다음 입력을 우선순위에 따라 확인한다.

1. 사용자가 원하는 작업 모드
2. 공식 공고문과 참가 자격
3. 평가항목과 배점
4. 제출양식, 분량, 파일 형식, 마감
5. 아이디어 또는 기존 초안
6. 근거자료와 출처
7. 유지해야 할 용어, 수치, 문체

정보가 부족해도 안전하게 진행할 수 있으면 가정을 명시하고 진행한다. 결과를 크게 바꾸는 정보가 없으면 사용자에게 확인한다.

## 5. 작업 모드

### 설계

아이디어와 공고문을 바탕으로 제안 핵심문, 평가항목별 주장 지도, 목차, 자료 보강 목록을 만든다.

### 초안

확정된 주장 지도와 근거만 사용해 기획서 초안을 작성한다.

### 진단

원문을 수정하지 않고 결함, 원문 근거, 심각도, 수정 방향을 보고한다.

### 재작성

사용자가 허용한 범위에서 사실과 의도를 보존하며 문장 또는 구조를 고친다. 새 사실을 추가하지 않는다.

### 최종 검수

제출조건, 평가항목 대응, 근거 통합, 용어·수치 일관성, 제목과 첫 문단의 이해도를 검사한다.

## 6. 글쓰기 평가 모델

15개 기준을 일곱 개의 검수 패스로 운영한다.

| 검수 패스 | 포함 기준 | 핵심 검사 |
| --- | --- | --- |
| 평가 정렬 | 신규 상위 단계 | 평가항목·배점·제출조건과 원고 근거를 연결하는가 |
| 30초 이해 | 후킹, 명료성, 강조 구조, 기억성 | 제목과 첫 문단만으로 문제·대상·제안·효과를 재구성할 수 있는가 |
| 논증 전개 | 논리 흐름, 정보 전진성, 근거 통합 | 각 문단이 앞에서 생긴 질문에 답하고 주장–근거–해석을 연결하는가 |
| 선택 설득 | 심사 결정 설득력 | 시급성–기존 공백–작동원리–적합성–실행 신뢰성–측정 변화가 제안의 선택 이유로 이어지는가 |
| 작동 구체화 | 구체성, 장면화 | 행위자·행동·대상·순서·결과와 실제 사용 장면이 보이는가 |
| 문장 설계 | 정보밀도, 인지부하, 어조 | 문장마다 기능이 있고 한 문장에 과도한 주장을 넣지 않으며 주장 강도가 적절한가 |
| 심사 지원 | 심사 편의성, 문서 일관성 | 채점 근거를 빨리 찾을 수 있고 용어·수치·단위가 일관되는가 |

### 용어 보정

- `후킹`은 클릭을 유도하는 표현이 아니라 제안의 정체를 즉시 밝히는 `초기 식별성`으로 정의한다.
- `정보밀도`는 문장에 사실을 많이 넣는 것이 아니라 각 문장이 필요한 기능을 수행하는 `문장 기능성`으로 정의한다.
- `장면화`는 감성 서사가 아니라 사용 전후의 행동과 서비스 반응을 보여주는 `작동 시나리오`로 정의한다.
- `어조`는 자신감의 정도보다 사실·추정·목표·가정을 구분하는 `주장 강도 보정`으로 정의한다.

## 7. 제안서 논리 골격

모든 제안에서 다음 핵심 요소를 먼저 고정한다.

1. `대상`: 누가 문제를 겪는가
2. `상황`: 언제, 어디서 문제가 발생하는가
3. `문제`: 관찰 가능한 불편·손실·위험은 무엇인가
4. `원인`: 왜 기존 방식으로 해결되지 않는가
5. `개입`: 누가 무엇을 어떻게 바꾸는가
6. `작동원리`: 개입이 어떤 중간 변화를 거쳐 결과를 만드는가
7. `결과`: 무엇이 얼마나 달라지는가
8. `측정`: 변화 여부를 어떤 지표와 시점으로 확인하는가
9. `실행`: 인력·예산·일정·협력자·위험 대응은 무엇인가
10. `차별성`: 기존 대안과 무엇이 다르고 왜 더 적합한가

각 문단에는 하나의 중심 질문을 지정하고 `질문 → 답 → 근거 → 의미 → 다음 질문`의 전진 구조를 검사한다.

## 8. 예상 Skill 구조

개발 위치는 플랜 승인 후 확정한다. 현재 작업 폴더에서 개발할 경우 예상 구조는 다음과 같다.

```text
write-competition-proposals/
├── SKILL.md
├── agents/
│   └── openai.yaml
└── references/
    ├── proposal-workflow.md
    ├── writing-rubric.md
    ├── evidence-integrity.md
    ├── korean-rewriting-patterns.md
    └── output-contracts.md
```

### 파일별 책임

#### `SKILL.md`

- 트리거가 되는 작업을 frontmatter `description`에 명시
- 입력 확인, 모드 선택, 핵심 워크플로, 중단 조건을 명령형으로 기술
- 필요한 참조 파일을 언제 읽을지 안내
- 사실 생성 금지와 원문 보존 규칙 명시

#### `agents/openai.yaml`

- `display_name`, `short_description`, `default_prompt` 제공
- 작성 전에 `skill-creator/references/openai_yaml.md`의 전체 지침 확인
- Skill 내용이 바뀌면 메타데이터의 일치 여부 재검사

#### `references/proposal-workflow.md`

- 공고문 분석
- 제안 핵심문 생성
- 평가항목별 주장 지도
- 문단 질문–답변 구조
- 초안·재작성·최종 검수 절차

#### `references/writing-rubric.md`

- 15개 글쓰기 기준의 조작적 정의
- 통과·부분 통과·실패 조건
- 흔한 실패 사례와 최소 수정 예시
- 기준 간 충돌 처리 규칙

#### `references/evidence-integrity.md`

- 사실·해석·추정·목표·가정 구분
- 통계의 출처·시점·모집단·단위 확인
- 주장–근거–해석 연결
- 대상·용어·수치·단위·기간 일관성 원장
- 근거가 없을 때 사용하는 표시 규칙

#### `references/korean-rewriting-patterns.md`

- 추상명사를 행위자–행동–결과로 변환
- 긴 문장 분리
- 주어와 실행주체 복원
- 중복 서술 제거
- 과장 표현과 모호한 수식어 조정
- 공모전 문체에 맞는 전후 예시

#### `references/output-contracts.md`

- 설계, 진단, 재작성, 최종 검수별 출력 형식
- 평가항목 근거표
- 문단 질문–답변 지도
- 우선순위 결함 보고서
- 일관성 원장
- 제출 전 체크리스트

### V1에서 만들지 않을 리소스

- `README.md`, 설치 가이드, 변경 이력 등 보조 문서
- 범용 템플릿을 가장한 고정 공모전 양식
- 검증되지 않은 점수 예측 스크립트
- 효과가 입증되지 않은 한국어 가독성 수치 계산기

V1은 `references/` 중심으로 구현한다. 반복 작업에서 명확한 필요가 확인될 때만 수치·용어 대조 스크립트를 추가한다.

## 9. 기본 출력 계약

작업 모드와 사용자 요청에 따라 필요한 항목만 출력하되, 전체 진단의 표준 결과는 다음 순서를 사용한다.

1. 한 문장 회상본
2. 30초 요약
3. `P0` 제출 위험
4. 가장 점수에 큰 영향을 주는 `P1` 문제
5. 평가항목별 근거 지도
6. 문단 질문–답변 흐름
7. 15개 글쓰기 기준 진단
8. 수정안 또는 재작성본
9. 대상·용어·수치·단위 일관성 원장
10. 요청받았거나 제출을 막는 필수 확인 정보

각 문제는 가능한 한 다음 형식을 따른다.

```text
[심각도 / 결함 종류 / 평가 기준]
원문: 문제가 발생한 실제 문장 또는 위치
판정: 왜 심사자의 이해나 채점에 문제가 되는가
조치: 삭제, 이동, 분리, 구체화, 근거 보강 중 무엇이 필요한가
수정안: 원문에 없는 사실을 추가하지 않은 예시
```

#### 고정 4회 교차검증 피드백 루프

최종 구현과 정적 검증을 마친 뒤 정확히 네 회차를 순차 실행한다. 네 회차를 병렬로 실행하지 않는다. 각 회차는 이전 대화가 전달되지 않은 새 하위 에이전트로 시작하고 현재 사용 중인 모델과 동일한 모델을 사용한다.

각 회차에서 다음 순서를 지킨다.

1. 새 에이전트에 Skill 경로와 원자료, 실제 사용자 형태의 과제만 제공한다.
2. 의도한 답, 기존 진단, 직전 에이전트의 피드백은 제공하지 않는다.
3. 에이전트의 실제 산출물을 먼저 받는다.
4. 해당 실행에서 발견한 누락·과잉·모호성·추가 또는 제거 후보에 대한 피드백을 받는다.
5. 주 에이전트가 피드백을 `plan.md`, Skill 범위, 근거 무결성, 이전 회귀 결과와 대조해 채택·기각·보류한다.
6. 채택한 피드백만 Skill에 반영하고 정적 검증과 관련 회귀 테스트를 다시 실행한다.
7. 수정된 현재 상태를 다음 회차의 새로운 에이전트가 검증하게 한다.

회차별 초점은 테스트 누출을 막기 위해 에이전트에게 정답 형태로 알려주지 않는다. 주 에이전트는 다음 관점이 네 회차 전체에서 다뤄졌는지 사후 확인한다.

- 1회차: 설계·초안 생성과 동적 평가기준 반영
- 2회차: 진단 전용 모드, 결함 분류, 근거 무결성
- 3회차: 재작성 보존성, 한국어 명료성, 일관성·인지부하
- 4회차: 최종 검수, 범위 과잉, 트리거 오작동, 전체 회귀

한 회차에서 피드백이 없더라도 회차를 생략하거나 합치지 않는다. 네 번째 수정 후 전체 정적 검증과 대표 회귀 테스트를 다시 통과해야 교차검증을 완료한 것으로 본다.

| 근거 무결성 | 새 통계·사례·성과를 발명하지 않으며 불확실성을 표시함 |
| 심사 편의성 | 평가항목별 근거와 원고 위치를 한 표에서 확인 가능 |
| 일관성 | 대상·용어·수치·단위·기간의 충돌을 보고함 |
| 재작성 보존성 | 원문 사실과 의도를 유지하고 추가한 가정을 표시함 |
| 출력 효율 | 동일 지적을 여러 섹션에서 반복하지 않고 우선순위가 명확함 |
| 구조 유효성 | 공식 `quick_validate.py`를 통과함 |

## 13. 예상 위험과 대응

### 과도한 만능화

공모전 종류마다 논리가 다르므로 V1을 서면형 제안서에 한정하고 공식 평가표를 최우선으로 한다.

### 진단 항목의 중복

한 문제를 여러 기준에서 반복하지 않는다. 주된 기준 하나에 배치하고 영향을 받는 보조 기준만 표시한다.

### 정보밀도와 인지부하의 충돌

문장 기능은 유지하되 한 문장에 하나의 중심 명제를 둔다. 정보 삭제보다 문장과 문단의 재배치를 우선한다.

### 후킹을 과장으로 오해

제목과 첫 문단은 자극성이 아니라 문제·대상·개입·변화를 빠르게 식별시키는 역할로 제한한다.

### AI의 근거 보완 환각

근거가 없으면 생성하지 않고 빈칸과 확인 질문을 출력하도록 강제한다. 사실 추가가 필요한 재작성은 사용자 자료 또는 검증 가능한 출처가 있을 때만 수행한다.

### 평가 점수의 허위 정밀성

총점 예측 대신 항목별 충족도, 원문 근거, 심각도와 개선 우선순위를 제공한다.

### Skill 본문의 비대화

세부 예시와 루브릭은 참조 파일로 이동하고 `SKILL.md`에는 선택과 실행에 필요한 절차만 유지한다.

## 14. 구현 전 확인할 결정사항

1. 최종 Skill 이름을 `write-competition-proposals`로 확정할지
2. 개발 위치를 현재 폴더 아래로 할지, 바로 `C:\Users\CKIRUser\.codex\skills`에 만들지
3. 첫 파일럿에 사용할 실제 공고문과 초안이 있는지
