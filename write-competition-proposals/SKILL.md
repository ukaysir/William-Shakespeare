# Write Competition Proposals

## 목적

심사자가 제목과 첫 문단에서 문제–대상–제안–효과를 파악하고, 공식 평가항목별 채점 근거를 빠르게 찾을 수 있는 한국어 서면형 공모전 기획서를 설계·작성·진단·재작성하라.

문장 미화보다 제출 적합성, 평가 대응, 논리, 근거 무결성, 심사 편의성을 우선하라. 수상이나 점수를 보장하지 마라.

## 입력 우선순위

가능한 자료를 다음 순서로 확인하라.

1. 사용자가 요청한 작업과 수정 범위
2. 최신 공식 공고문과 수정 공지
3. 공식 평가표·배점·과락·동점 규칙
4. 제출양식·분량·파일·마감·첨부 조건
5. 아이디어 또는 기존 초안
6. 사용자 제공 근거와 확정 용어·수치
7. 검증 가능한 외부 근거

공식 자료가 없으면 안전하게 가능한 범위만 수행하고 `일반 보조 기준`이라고 밝혀라. 공식 평가항목이나 예상 점수를 만들어내지 마라.

## 작업 모드 선택

사용자의 요청에 맞는 가장 좁은 모드를 선택하라.

| 모드 | 사용 조건 | 핵심 결과 |
| --- | --- | --- |
| 설계 | 아이디어를 구조화하거나 목차를 만들 때 | 제안 핵심, 평가 지도, 목차, 정보 공백 |
| 초안 | 제공 자료로 새 원고를 쓸 때 | 양식에 맞춘 초안과 확인 필요 항목 |
| 진단 | 문제점·평가만 요청할 때 | 원문 보존 진단, 근거, 우선순위, 조치 |
| 재작성 | 문장·문단·전체 개선을 요청할 때 | 사실 보존 수정본과 핵심 변경 |
| 최종 검수 | 제출 직전 확인할 때 | 제출 위험, 평가 대응, 근거·일관성 감사 |

진단만 요청받으면 허가 없이 전체를 재작성하지 마라. 여러 모드가 필요하면 `진단 → 허용 범위 재작성 → 최종 검수` 순서로 수행하라.

## 핵심 워크플로

### 1. 작업 계약 잠그기

요청 모드, 대상 문서, 수정 허용 범위, 유지할 사실과 형식을 확인하라. 결과를 실질적으로 바꾸지 않는 누락은 가정을 밝히고 계속하라.

### 2. 공고문 잠그기

공모 목적, 분야, 참가 자격, 마감, 제출조건, 제외·실격 조건을 먼저 추출하라. 평가항목명·세부기준·배점·과락을 원형 그대로 표로 잠가라.

자격·마감·필수 첨부·공식 판본이 충돌하면 `P0 / COMPLIANCE_RISK`로 먼저 보고하라.

### 3. 제안 핵심 잠그기

다음 열 요소를 원문과 자료에서 채워라.

`대상 → 상황 → 문제 → 원인 → 개입 → 작동원리 → 결과 → 측정 → 실행 → 차별성`

정보가 없으면 추측하지 말고 구체적인 확인 표식을 남겨라. 이 요소로 한 문장 제안을 만들고 제목과 첫 문단만으로 문제·대상·제안·효과를 재구성할 수 있는지 검사하라.

### 4. 평가항목을 역설계하기

각 공식 평가항목에 대해 다음을 연결하라.

`심사 질문 → 핵심 답 → 필요한 근거 → 현재 근거 → 원고 위치 → 상태 → 조치`

높은 배점과 과락 항목을 우선하라. 공식 양식의 필드와 순서를 보존하고, 각 필드 안에서 결론을 먼저 배치하라.

### 5. 공백과 표현 문제 분리하기

가장 근본적인 결함 하나를 지정하라.

- `CONTENT_GAP`: 주장·계획·근거·지표 자체가 없음
- `WRITING_GAP`: 정보는 있으나 위치·표현·연결이 약함
- `EVIDENCE_RISK`: 출처·수치·인과·주장 상태가 불명확함
- `COMPLIANCE_RISK`: 자격·서식·마감·권리·필수조건 위험

심각도를 별도로 지정하라.

- `P0`: 제출·사실성·핵심 이해를 무너뜨림
- `P1`: 공식 평가항목을 직접 약화함
- `P2`: 탐색 속도·가독성·완성도를 낮춤

같은 문제를 여러 기준에서 반복하지 말고 주된 결함에 상세 설명을 모아라.

### 6. 구조를 설계하고 작성하기

공식 양식이 있으면 순서를 유지하라. 자유 형식이면 공고의 평가 질문에 맞춰 다음 기본 흐름을 조정하라.

`제안 요약 → 문제·필요성 → 원인·기존 공백 → 제안과 작동 방식 → 차별성 → 실행 → 효과·측정 → 지속·확산`

각 문단을 `독자 질문 → 첫 문장의 답 → 근거 → 의미 → 다음 질문`으로 설계하라. 문단 첫 문장에 결론을 두고, 행위자–행동–조건–결과를 드러내라.

### 7. 근거와 주장 강도 감사하기

핵심 문장을 사실, 해석, 추정, 목표, 계획, 가정, 미확정으로 구분하라. 근거보다 강한 동사를 낮춰라.

통계는 출처·시점·대상·분모·단위를 확인하고 `주장 → 근거 → 해석 → 설계 함의`로 연결하라. 문제의 존재를 보여주는 자료를 특정 해법의 효과 증거로 오용하지 마라.

### 8. 심사자 관점으로 검수하기

다음 순서로 검사하라.

1. 자격·제출조건·권리
2. 공식 평가항목별 근거 회수 가능성
3. 문제–원인–개입–결과–측정의 연결
4. 제목·첫 문단의 30초 이해
5. 14개 글쓰기 기준
6. 대상·용어·수치·단위·기간·주체 일관성

에이전트의 이해도 검사는 실제 독자 검사의 대리 지표라고 밝혀라. 가능하면 사용자가 실제 독자에게 자기 말로 설명하게 하는 검사를 권하라.

## 사실 무결성 규칙

다음을 반드시 지켜라.

- 새 통계·인터뷰·성과·협의·승인·예산·성능을 사실처럼 만들지 마라.
- 목표를 성과로, 계획을 완료 사실로, 가정을 수요 사실로 바꾸지 마라.
- 원문의 수치·대상·기간을 조용히 변경하지 마라.
- 비교 근거 없이 `최초`, `유일`, `완벽`, `획기적`을 쓰지 마라.
- 근거가 없으면 `[무엇을 확인해야 하는지]`가 드러나는 표식을 사용하라.
- 법률·의료·재무·기술 타당성을 전문가 확인 없이 확정하지 마라.

사용자가 웹 조사를 요청하거나 최신·고위험 사실이 필요하면 검증 가능한 원문을 찾아 출처를 가까이 표시하라. 사용자 초안의 민감정보를 외부 검색어로 사용하지 마라.

## 출력 규칙

결과부터 제시하고 작업 과정을 길게 설명하지 마라. 요청 모드에 필요한 출력만 선택하라.

진단 항목은 가능한 한 다음 구조를 사용하라.

```text
[심각도 / 결함 종류 / 공식 항목 또는 글쓰기 기준]
원문: 실제 문장 또는 위치
판정: 심사자가 답하지 못하는 질문
영향: 평가 또는 이해에 미치는 영향
조치: 필요한 최소 행동
수정 예: 새 사실을 추가하지 않은 예시
```

근거 없는 총점과 수상 확률을 제공하지 마라. 글쓰기 충족도는 필요할 때만 `0=부재·모순`, `1=부분`, `2=명확`으로 표시하고 반드시 원문 근거를 붙여라.

## 중단 조건

다음 정보가 결과를 실질적으로 바꾸면 확인을 요청하라.

- 공식 판본이 자격·마감·평가표에서 충돌함
- 지원 분야 선택에 따라 제안 구조가 달라짐
- 원문의 핵심 아이디어를 바꿔야만 요청을 충족함
- 민감정보, 비공개 자료, 법적 확정 판단이 필요함
- 사실 추가 없이는 사용자가 요구한 효과·성과를 쓸 수 없음

확인 전에도 수행할 수 있는 공고 정리, 원문 진단, 공백 표시는 계속하라.

## 완료 기준

다음을 모두 확인한 뒤 완료하라.

- 요청한 모드의 산출물이 앞에 있음
- 공식 조건과 자체 보조 기준이 구분됨
- 모든 공식 평가항목에 근거 또는 명시적 공백이 있음
- 제목과 첫 문단에서 문제·대상·제안·효과를 재구성할 수 있음
- 핵심 논리의 빠진 연결이 숨겨지지 않음
- 새 사실이나 근거를 발명하지 않음
- 대상·용어·수치·단위·기간·주체의 충돌을 보고함
- 가장 중요한 다음 행동이 우선순위로 제시됨
