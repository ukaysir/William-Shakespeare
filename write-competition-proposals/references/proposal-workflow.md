# 공모전 제안서 승리 워크플로

## 1. 작업 모드 결정

사용자 동사에 맞춰 산출물을 선택하라.

| 모드 | 사용자 요청 | 첫 산출물 |
| --- | --- | --- |
| 설계 | 아이디어 구조화·기획 | 확정 논리사슬 |
| 초안 | 처음부터 작성 | 제출용 원고 |
| 진단 | 문제·논리·점수 손실 확인 | 가장 치명적인 단절 |
| 재작성 | 문장·문단·전체 개선 | 완성 수정본 |
| 최종 검수 | 제출 직전 점검 | 채택을 막는 잔여 결함과 교체문 |

여러 모드가 결합되면 `논리사슬 확정 → 작성·재작성 → 압승 감사` 순서로 수행하라.

## 2. 공모전 승리 조건 추출

최신 공식 공고, 정정 공지, 평가표, 제출양식에서 다음을 잠가라.

| 구분 | 추출 항목 |
| --- | --- |
| 목적 | 주최기관이 해결하거나 발굴하려는 핵심 변화 |
| 대상 | 참가 자격, 지원 분야, 수혜 대상 |
| 심사 | 항목명, 세부 질문, 배점, 과락, 동점 기준 |
| 제출 | 마감, 분량, 파일, 양식, 필수 첨부 |
| 제외 | 중복, 기수상, 권리, 금지 분야 |

공식 평가항목을 다음처럼 변환하라.

`공식 문구 → 심사자가 답할 질문 → 최고점 판정 → 필요한 증거`

예:

| 공식 항목 | 심사 질문 | 최고점 판정 | 필요한 증거 |
| --- | --- | --- | --- |
| 실현가능성 | 누가 실제로 끝낼 수 있는가 | 주체·자원·일정·절차·완료 기준이 맞물린다 | 역할표, 일정, 예산 근거, 실행 경험 |
| 효과성 | 무엇이 얼마나 달라지는가 | 개입부터 성과지표까지 경로가 측정 가능하다 | 기준선, 중간지표, 목표, 측정법 |

## 3. 채택명제 확정

다음 요소를 하나의 문장에 통합하라.

`대상 + 핵심 문제 + 구조적 원인 + 차별적 개입 + 작동 메커니즘 + 공모 목적에 직결되는 성과`

형식:

> 심사위원회는 `[대상]`의 `[문제]`를 만드는 `[원인]`을 `[개입과 메커니즘]`으로 바꾸어 `[성과]`를 실현하는 본 제안을 선택해야 한다.

채택명제가 두 개면 제안이 둘인 것이다. 부가 기능을 제거하거나 하나의 메커니즘 아래 통합하라.

## 4. 논리사슬 확정

[logic-chain.md](logic-chain.md)의 `확정 논리사슬 표`를 작성하라. 초안 전에 다음 세 가지를 반드시 잠근다.

1. `선택 논리`: 왜 지금 이 문제이며 왜 이 제안인가
2. `변화 논리`: 제안이 어떤 단계로 결과를 만드는가
3. `득점 논리`: 각 연결이 어느 평가항목의 어떤 판정을 만드는가

연결이 비면 웹 조사, 사용자 자료 재탐색, 설계 구체화, 주장 교체 순서로 해결하라. 빈칸을 추상 문장으로 덮지 마라.

## 5. 근거를 공격 무기로 배치

근거마다 다음 네 질문에 답하라.

1. 이 근거는 어떤 명제를 증명하는가
2. 문제·원인·해법·실행·성과 중 어디에 쓰이는가
3. 근거와 명제를 잇는 논거는 무엇인가
4. 어느 평가항목의 점수를 만드는가

근거가 많아도 같은 주장만 반복하면 공격력이 늘지 않는다. 핵심 연결을 하나씩 닫는 근거를 우선하라.

## 6. 구조 배치

### 공식 양식

필드와 순서를 유지한다. 각 필드 안에 다음을 배치하라.

1. 심사 질문에 대한 판정 문장
2. 핵심 근거
3. 근거의 의미
4. 작동 방식 또는 실행 증명
5. 해당 항목의 선택 결론

### 자유 형식

1. 채택 결론
2. 문제·대상·시급성
3. 구조적 원인
4. 기존 방식의 공백
5. 개입과 작동 메커니즘
6. 비교우위
7. 실행 증명
8. 산출·성과·측정
9. 공모 목적과 평가항목 회수
10. 최종 채택 결론

## 7. 문단 작성

각 문단을 다음 구조로 완성하라.

`판정 문장 → 근거 → 논거 → 설계·실행 함의 → 다음 판단`

문단 첫 문장만 이어 읽었을 때 채택명제가 완성되어야 한다. 문단이 새 명제·근거·논거·비교·실행·측정 중 아무것도 추가하지 않으면 삭제하라.

## 8. 반론 제압

실제 심사 결정을 막는 반론만 고른다. 반론을 별도 고백문으로 쓰지 말고 해당 평가항목 안에서 다음 중 하나로 해소하라.

- 비교자료로 해소
- 설계 선택으로 해소
- 실행주체와 책임으로 해소
- 단계별 검증으로 해소
- 측정지표로 해소
- 대체 절차로 해소

형식:

> `[선택을 막는 의문]`은 `[설계·근거·실행 장치]`로 해소된다. 이 장치는 `[평가기준]`에서 `[심사 결론]`을 만든다.

## 9. 표현 강화

- 행위자를 문장 앞에 둔다.
- 명사형 계획을 행동 동사로 바꾼다.
- `필요하다` 다음에는 무엇을 선택해야 하는지 쓴다.
- 수치 다음에는 그 수치가 어떤 설계를 정당화하는지 쓴다.
- 비교 다음에는 심사자가 내려야 할 판정을 쓴다.
- 계획 다음에는 담당·시점·완료 기준을 쓴다.
- 결론은 실행과 성과로 닫는다.

## 10. 최종 압박 검사

### 논리

- 채택명제가 하나인가
- 원인과 개입이 직접 연결되는가
- 작동 메커니즘에 중간 변화가 있는가
- 산출과 성과가 구분되는가
- 정방향과 역방향 사슬이 일치하는가

### 경쟁

- 기존 방식의 결정적 공백이 구체적인가
- 같은 기준에서 비교했는가
- 제안의 승부처가 1~3개로 선명한가
- 반론이 설계·근거·실행으로 해소되는가

### 득점

- 모든 공식 항목에 판정 문장이 있는가
- 높은 배점 항목의 근거가 앞쪽에 있는가
- 심사자가 근거를 즉시 인용할 수 있는가
- 마지막 결론이 공모 목적과 평가항목을 회수하는가

### 문장

- 첫 세 문장이 문제·대상·개입·성과를 말하는가
- 능동형과 단정형이 지배하는가
- 빈 형용사와 관성적 완곡어가 제거됐는가
- 마지막 문장이 선택 이유를 못 박는가

## 11. 자료가 적을 때도 전진하기

- 공고문이 없으면 사용자 요청과 문서 목적에서 판정 질문을 만들고 `보조 판단축`으로 사용하라.
- 평가표가 없으면 공모 목적·양식·과거 공식 공고를 조사해 후보 질문을 만들되 공식 항목으로 단정하지 마라.
- 수치가 없으면 구조적 인과, 실행 절차, 측정 설계를 먼저 강화하라.
- 비교자료가 없으면 확인된 기능 차이를 쓰고 즉시 조사할 비교 기준을 내부 작업표에 남겨라.
- 결과를 바꾸는 단 하나의 선택이 필요할 때만 사용자에게 짧게 질문하라. 나머지는 가장 강한 합리적 구조로 계속 작성하라.
