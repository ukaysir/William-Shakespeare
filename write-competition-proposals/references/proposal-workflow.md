## 1. 작업 모드 결정

사용자 요청에 맞춰 첫 산출물을 정하라.

| 모드 | 요청 | 첫 산출물 |
| --- | --- | --- |
| 설계 | 아이디어 구조화·기획 | 7부 구조 설계안 |
| 초안 | 처음부터 작성 | 제목과 7부 제출 원고 |
| 진단 | 문제·논리·점수 손실 확인 | 가장 치명적인 결함 |
| 재작성 | 문장·문단·전체 개선 | 7부 구조 완성 수정본 |
| 최종 검수 | 제출 직전 점검 | 잔여 결함·교체문·제출 판정 |

여러 모드가 결합되면 내부적으로 `승리 조건 확인 → 7부 구조 설계 → 작성·재작성 → 최종 검수` 순서로 수행하라.

## 2. 승리 조건 추출

최신 공식 공고, 정정 공지, 평가표, 제출양식에서 다음을 잠가라.

| 구분 | 추출 항목 |
| --- | --- |
| 목적 | 주최기관이 해결하거나 발굴하려는 변화 |
| 대상 | 참가 자격, 지원 분야, 수혜 대상 |
| 심사 | 항목명, 세부 질문, 배점, 과락, 동점 기준 |
| 제출 | 마감, 분량, 파일, 양식, 필수 첨부 |
| 제외 | 중복, 기수상, 권리, 금지 분야 |

공식 평가항목을 다음처럼 변환하라.

`공식 문구 → 심사 질문 → 필요한 판단 → 필요한 근거 → 7부 구조의 위치`

평가항목의 배점은 근거의 양과 구체성을 조절하는 기준으로 사용하라. 배점에 따라 7부 구조의 순서를 바꾸지 마라.

## 3. 핵심 결론 확정

다음 요소를 한 문장에 통합하라.

`실행주체 + 대상 + 핵심 문제 + 핵심 원인 + 해결 방안`

형식:

> `[실행주체]`는 `[대상]`이 겪는 `[핵심 문제]`의 `[핵심 원인]`을 해결하기 위해 `[해결 방안]`을 실행해야 한다.

핵심 결론이 둘이면 제안이 둘인 것이다. 부가기능을 제거하거나 하나의 해결 방안 아래 통합하라.

## 4. 문제–원인–해결 대응표 작성

초안 전에 다음 표를 완성하라.

| 문제와 손실 | 원인 | 대응 해결 방안 | 실행 조건 | 직접 효과 |
| --- | --- | --- | --- | --- |
|  |  |  |  |  |

다음을 지켜라.

- 현황은 벌어지는 일, 문제는 대상이 입는 손실, 원인은 손실이 반복되는 이유로 구분하라.
- 원인이 여러 개면 번호와 순서를 유지해 해결 방안을 일대일로 대응시켜라.
- 원인을 직접 바꾸지 않는 기능은 삭제하거나 부가기능으로 낮춰라.
- 해결 방안에서 기대 효과로 바로 뛰지 말고 실행 조건을 확인하라.

## 5. 유일한 7부 구조 설계

다음 구조만 사용하라.

| 순서 | 섹션 | 반드시 답할 질문 | 필수 내용 |
| ---: | --- | --- | --- |
| 1 | 제안 요약 | 무엇을 제안하는가 | 현재 상황, 핵심 문제, 핵심 결론, 실행 방향 |
| 2 | 문제 상황 | 누가 무엇 때문에 어려운가 | 대상, 구체적 손실, 지속 결과, 시급성 |
| 3 | 문제의 원인 | 왜 문제가 반복되는가 | 실질적·구조적 원인, 기존 대응의 한계 |
| 4 | 해결 방안 | 누가 무엇을 어떻게 하는가 | 실행주체, 대상, 행동, 운영방식, 원인별 대응 |
| 5 | 차별성과 실현 가능성 | 왜 다르고 실제로 가능한가 | 동일 기준 비교, 주체, 자원, 절차, 일정, 위험 대응, 완료 기준 |
| 6 | 기대 효과 | 무엇이 어떻게 달라지는가 | 직접·확장·장기 효과, 지표, 측정법, 시점 |
| 7 | 결론 | 왜 지금 선택해야 하는가 | 필요성, 원인 적합성, 실행 가능성, 핵심 효과, 선택 이유 |

공식 양식이 있으면 필드와 순서를 유지하고 각 필드에 위 일곱 기능 중 하나 이상을 배정하라. 빠진 기능이 없도록 하되 별도의 대체 구조를 만들지 마라.

## 6. 단일 문단 공식 적용

모든 문단을 다음 하나의 공식으로 작성하라.

`주장 → 설명 → 근거·사례 → 소결론`

| 단계 | 작성 내용 |
| --- | --- |
| 주장 | 문단의 중심 판단을 첫 문장으로 제시 |
| 설명 | 이유·원인·작동 방식·필요한 맥락을 설명 |
| 근거·사례 | 확인된 자료·사례·비교·설계·계획으로 뒷받침 |
| 소결론 | 근거의 의미를 해석하고 섹션 판단 또는 다음 단계로 연결 |

한 문단에 중심 주장을 하나만 둬라. 근거가 없는 문단은 확인 가능한 설계·실행계획으로 뒷받침하거나 주장 강도를 낮춰라. 문단 첫 문장만 이어 읽었을 때 7부 구조가 완성되어야 한다.

## 7. 차별성과 실현 가능성 작성

비교는 실제 승부처에 해당하는 기준 2~4개만 사용하라.

| 동일 기준 | 기존 방식 | 본 제안 | 소결론 |
| --- | --- | --- | --- |
| 대상 적합성/원인 대응/절차/시간/비용/측정 가능성 중 선택 |  |  |  |

실현 가능성은 다음 요소로 증명하라.

`실행주체 + 확보·조달 자원 + 단계별 절차 + 일정 + 위험과 보완책 + 완료 기준`

반론은 별도 구조로 만들지 말고 5부의 문단 안에서 위험과 보완책으로 처리하라.

## 8. 기대 효과 작성

효과를 다음 세 단계로 구분하라.

1. 직접 효과: 해결 방안 실행 직후 대상에게 생기는 변화
2. 확장 효과: 직접 효과가 행동·협업·접근성 등에 만드는 후속 변화
3. 장기 효과: 확장 효과가 유지될 때 나타나는 장기 변화

각 효과에 지표·측정법·시점을 붙여라. 해결 방안이 직접 만들 수 있는 변화부터 설명하고 장기 효과를 이미 달성한 사실처럼 쓰지 마라.

## 9. 평가항목 득점 지도 작성

공식 평가항목을 7부 구조에 연결하라.

| 공식 항목 | 배점 | 필요한 판단 | 근거 | 원고 섹션 |
| --- | ---: | --- | --- | --- |
|  |  |  |  |  |

공식 항목에 필요한 판단이 원고에서 즉시 확인되는지 검사하라. 별도의 평가항목별 반복 결론을 만들지 말고 해당 섹션의 주장과 소결론에 자연스럽게 반영하라.

## 10. 자료가 적을 때 처리

- 공고문이 없으면 사용자 요청과 문서 목적에서 보조 질문을 만들고 공식 기준과 구분하라.
- 수치가 없으면 구조적 원인, 실행 절차, 측정 설계를 먼저 구체화하라.
- 비교자료가 없으면 확인된 기능·절차 차이만 쓰고 최상급을 사용하지 마라.
- 기준선이 없으면 첫 실행 단계에서 기준선을 측정하도록 설계하라.
- 필수값 하나가 전체 결과를 바꿀 때만 사용자에게 짧게 확인하라.

## 11. 최종 순서 검사

- 제목에서 문제 또는 해결책이 보이는가
- 첫 문단에서 주제와 핵심 결론이 보이는가
- 7부 구조가 빠짐없이 같은 순서로 진행되는가
- 모든 문단이 단일 문단 공식을 따르는가
- 문제와 원인이 구분되는가
- 원인과 해결 방안이 일대일로 연결되는가
- 해결 방안의 실제 작동 모습이 보이는가
- 차별성과 실현 가능성이 구체적인가
- 기대 효과가 측정 가능한가
- 마지막 결론에 새로운 내용이 없는가
- 문단 첫 문장만 읽어도 전체 흐름이 이어지는가
