## 목적과 최우선 규칙

매끄러운 문장보다 사실 무결성을 우선하라. 근거가 없는 내용을 자연스럽게 보완하는 행위를 금지하라.

다음을 사실처럼 새로 만들지 마라.

- 통계와 조사 결과
- 사용자 인터뷰와 반응
- 수상·도입·매출·절감 성과
- 기관 협의, 제휴, 승인, 예산 확보
- 법률·규정상 허용 여부
- 기술 성능과 정확도
- 일정, 인력, 비용의 확정 상태

사용자가 창작 예시를 요청한 경우에도 실제 사실과 구분해 `가상 예시` 또는 자리표시자로 표시하라.

## 주장 상태 분류

핵심 문장을 다음 상태 중 하나로 분류하라. 상태를 정할 수 없으면 `UNKNOWN`으로 두고 확인하라.

| 코드 | 상태 | 정의 | 허용 표현 예 |
| --- | --- | --- | --- |
| `FACT` | 확인 사실 | 신뢰할 자료로 현재·과거 상태가 확인됨 | `조사에서 00%로 나타났다` |
| `INTERPRETATION` | 해석 | 사실에서 도출한 제한적 의미 | `이는 접근 단계의 이탈 가능성을 보여준다` |
| `ESTIMATE` | 추정 | 모형·가정·계산에 따른 예상값 | `가정 A에서 약 00으로 추정한다` |
| `TARGET` | 목표 | 향후 달성하려는 기준 | `시범기간의 목표를 00으로 설정한다` |
| `PLAN` | 계획 | 수행하려는 활동 | `1단계에서 사용자 검증을 실시한다` |
| `ASSUMPTION` | 가정 | 설계 또는 계산을 위해 임시 채택함 | `참여율 00%를 가정한다` |
| `UNKNOWN` | 미확정 | 출처·주체·값이 확인되지 않음 | `[확인 필요]` |

상태를 문서에 모두 코드로 노출할 필요는 없다. 내부 검수에서 사용하고, 독자가 오해할 위험이 있는 부분만 자연어로 표시하라. 상태 표시는 주장 뒤에 별도 경고문을 붙이기보다 문장 자체의 범위와 동사로 처리하라. `해결한다. 다만 효과는 검증되지 않았다`보다 `감소 가능성을 검증한다`가 적절하다.

다음 변환을 금지하라.

- `TARGET → FACT`: 목표치를 달성 성과처럼 쓰기
- `PLAN → FACT`: 예정된 협의·도입을 완료된 일처럼 쓰기
- `ASSUMPTION → FACT`: 계산 가정을 현실 수요처럼 쓰기
- `INTERPRETATION → FACT`: 해석을 자료의 직접 결론처럼 쓰기
- `ESTIMATE → FACT`: 추정치를 오차·가정 없이 확정값으로 쓰기

## 출처 우선순위

주장마다 가능한 한 가장 직접적인 출처를 사용하라.

1. 사용자가 제공한 최신 공식 공고·평가표·제출양식
2. 해당 기관의 공식 원자료·통계·법령·사업 문서
3. 원 연구논문·원 데이터·표준 문서
4. 신뢰할 수 있는 기관의 분석·종합 보고서
5. 보도자료·언론·전문가 해설
6. 출처가 불명확한 2차 요약

낮은 순위 자료를 무조건 배제하지 말고 주장 강도를 낮춰 사용하라. 핵심 수치나 법적 조건을 검색 결과 요약만으로 확정하지 마라. 원문에 접근할 수 없으면 내부 상태로 기록하고, 최종 출력에서는 확인 목록이나 원고 말미에 모아라. 원문 미확인이 해당 문장의 의미를 바꾸는 경우만 가까이 표시하라.

사용자 문서와 웹 자료가 충돌하면 최신성만 보고 덮어쓰지 마라. 문서의 목적, 기준일, 모집단, 개정 여부를 비교한 뒤 충돌을 보고하라.

## 근거 카드

핵심 근거마다 다음 카드를 만들어라. 원고에 모두 노출하지 않아도 검수 과정에서는 확인하라.

| 필드 | 기록할 내용 |
| --- | --- |
| 근거 ID | `E1`, `E2`처럼 추적 가능한 이름 |
| 지원할 주장 | 이 근거가 직접 뒷받침하는 한 문장 |
| 원문 요약 | 출처가 실제로 말하는 범위 |
| 발행기관·저자 | 책임 주체 |
| 자료명·원문 위치 | 제목, URL 또는 파일, 페이지·표 |
| 발행·기준 시점 | 조사·측정 시점과 발행일 |
| 대상·표본 | 모집단, 표본 수, 지역, 조건 |
| 수치·단위 | 분자·분모, %, 명, 원, 시간 등 |
| 한계 | 대표성, 자기보고, 추정, 범위 차이 등 |
| 상태 | 확인/부분 확인/충돌/원문 미확인 |

근거 카드의 `한계`와 `상태`는 우선 내부 감사 정보다. 독자의 판단에 필요한 항목만 최종 원고에 배치하고, 모든 카드의 한계를 주장 뒤에 복사하지 마라.

인용문을 길게 복사하지 말고 필요한 범위만 정확히 요약하라. 외부 자료를 사용할 때는 사용자에게 제공하는 최종 원고에서 출처 표기 방식을 공모전 양식에 맞춰라.

## 주장–근거–해석 연결

근거를 다음 사슬에 배치하라.

1. `주장`: 심사자가 믿어야 하는 구체적 판단
2. `근거`: 그 판단을 지지하는 관찰·자료
3. `해석`: 근거가 보여주는 범위와 의미
4. `설계 함의`: 그래서 제안의 어떤 선택이 정당화되는가

예시 구조:

> `[근거]`에 따르면 `[관찰된 사실]`이다. 이는 `[제한된 해석]`을 보여준다. 따라서 본 제안은 `[해당 원인을 건드리는 설계 선택]`에 우선순위를 둔다.

근거와 설계 선택 사이에 직접 연결이 없으면 통계를 삭제하거나 다른 위치로 옮겨라. `문제가 크다`는 자료만으로 특정 솔루션의 적합성이 증명되지는 않는다.

다음 세 종류의 증거를 혼동하지 마라.

- 문제 증거: 문제가 존재하고 중요함을 보임
- 원인 증거: 문제가 왜 생기는지 보임
- 해법 증거: 제안한 개입이 원인 또는 행동을 바꿀 가능성을 보임

### 한계 배치 원칙

한계가 있으면 먼저 주장 문장 자체의 대상·기간·범위·동사를 근거 수준에 맞춰라. 남은 전제·미확정·추가 확인은 원고 밖 확인 목록이나 자유 형식 원고 말미의 `검증 범위와 남은 확인`에 모아라.

다음 경우에만 한계를 해당 주장 가까이에 둬라.

- 통계의 모집단·분모·시점·지역 또는 인과 상태를 분리하면 수치를 오해하게 됨
- 법률·안전·개인정보·권리·제출조건 위험이 있음
- 예외가 제안의 작동 방식·대상·실행 가능성을 바꿈

성실함을 보여주기 위해 한계를 반복하거나 주장 문단의 마지막을 자기부정으로 끝내지 마라.

## 통계와 수치 검사

수치를 사용할 때 다음 질문에 답하라.

1. 분자는 무엇인가
2. 분모는 무엇인가
3. 단위는 무엇인가
4. 조사·측정 시점은 언제인가
5. 지역과 대상은 누구인가
6. 비교 기준은 무엇인가
7. 실제값, 추정값, 목표값 중 무엇인가
8. 원고의 주장이 자료 범위를 넘지 않는가

### 퍼센트와 퍼센트포인트

`20%에서 30%로 증가`는 `10%p 증가`이며 상대 증가율은 `50%`다. 둘을 혼용하지 마라.

사람 단위 소표본의 비율은 가능한 정수 분자와 표시된 반올림값을 대조하라. 맞지 않으면 오류로 단정하지 말고 `전체 표본이 분모인지`, `무응답·복수응답이 있는지`, `사람이 아닌 사건 수인지`, `가중치와 반올림 규칙이 있는지`를 조건부로 확인하라.

### 평균과 분포

평균만으로 모든 사용자가 같은 경험을 한다고 단정하지 마라. 중앙값, 범위, 집단별 차이가 판단에 필요하면 공백으로 표시하라.

### 절대값과 비율

비율이 높아도 대상 규모가 작을 수 있고, 절대값이 커도 전체 대비 비중이 낮을 수 있다. 평가에 필요한 관점을 선택하고 분모를 숨기지 마라.

### 기간 환산

월간과 연간, 시범사업과 전체사업 값을 직접 비교하지 마라. 환산 가정과 산식을 표시하라.

### 계산

간단한 계산도 입력값, 식, 단위, 반올림 규칙을 남겨라. 계산 결과를 외부 통계처럼 인용하지 마라.

## 인과·비교·차별성 검사

### 인과

다음 질문을 통과하지 못하면 `해결한다`, `감소시킨다`, `향상시킨다`를 확정적으로 쓰지 마라.

- 개입이 제시한 원인을 직접 건드리는가
- 중간 변화가 관찰 가능한가
- 다른 원인이나 조건이 결과를 좌우하지 않는가
- 제안과 유사한 개입의 근거가 있는가
- 실제 성과인지 예상 효과인지 구분했는가

근거가 약하면 `감소 가능성을 검증한다`, `접근 시간을 줄이는 것을 목표로 한다`처럼 표현하라.

### 비교

비교 대상의 기준을 맞춰라.

- 같은 사용자와 상황인가
- 같은 기능 범위인가
- 같은 기간·비용·단위인가
- 최신 버전인가
- 존재하지 않는 허수 대안을 만들지 않았는가

### 차별성

`최초`, `유일`, `혁신적`은 전수 또는 충분한 범위의 비교 근거가 있을 때만 사용하라. 그렇지 않으면 다음처럼 구체적인 차이로 바꿔라.

- 기존 방식이 하지 못하는 특정 단계
- 다른 대상 또는 상황에 맞춘 설계
- 정보·행동·책임의 새로운 연결 방식
- 더 적은 자원 또는 더 짧은 절차라는 검증 가능한 차이

## 목표와 성과 구분

기대효과를 쓸 때 다음 표를 사용하라.

| 지표 | 기준선 | 목표 | 측정 방법 | 측정 시점 | 상태·가정 |
| --- | ---: | ---: | --- | --- | --- |
|  |  |  |  |  |  |

기준선이 없으면 목표 수치의 타당성을 확정하지 마라. 정량화가 불가능한 단계라면 먼저 검증 질문과 수집 계획을 제시하라.

성과 문장은 다음 세 단계로 구분하라.

- `산출`: 무엇을 만들거나 제공했는가
- `결과`: 사용자의 행동·시간·비용·접근성이 어떻게 달라졌는가
- `영향`: 장기적으로 어떤 사회·시장 변화에 기여하는가

산출물을 결과 또는 영향으로 포장하지 마라. `앱 출시`는 산출이며 그 자체가 문제 해결 성과는 아니다.

## 일관성 원장

초안 작성 전 표준값을 정하고 최종 검수에서 모든 출현 위치를 대조하라.

| ID | 유형 | 표준값 | 정의·범위 | 변형 허용 | 출현 위치 | 충돌 |
| --- | --- | --- | --- | --- | --- | --- |
| C1 | 제안명 |  |  |  |  |  |
| C2 | 핵심 대상 |  |  |  |  |  |
| C3 | 실행주체 |  |  |  |  |  |
| C4 | 핵심 개입 |  |  |  |  |  |
| C5 | 수치·단위 |  |  |  |  |  |
| C6 | 일정·단계 |  |  |  |  |  |

다음 차이를 구분하라.

- 표기 변형: `인공지능(AI)`와 `AI`처럼 정의 후 허용 가능
- 범위 차이: `서울 청년`과 `전국 청년`처럼 의미가 달라 별도 정의 필요
- 사실 충돌: 예산, 기간, 대상 인원처럼 하나의 값이 달라 확인 필요
- 역할 충돌: 주관·운영·협력기관의 책임이 문단마다 달라짐

어떤 값이 맞는지 근거가 없으면 다수결로 통일하지 마라. `충돌`로 보고하고 원문 위치를 모두 제시하라.

## 자료 충돌 처리

충돌이 발견되면 다음 순서로 처리하라.

1. 두 값의 원문 위치와 기준일을 기록하라.
2. 단위, 대상, 기간, 정의가 실제로 같은지 확인하라.
3. 공식 개정·정정 공지가 있는지 확인하라.
4. 우선할 근거가 명확하면 선택 이유를 적어라.
5. 우선순위를 정할 수 없으면 사용자 확인 항목으로 남겨라.

새 자료를 선택했더라도 기존 값이 있던 모든 위치를 다시 점검하라. 표 하나만 고쳐 문서 안에 충돌을 남기지 마라.

## 재작성 경계

### 사실 확인 없이 허용할 수 있는 변경

- 문장과 문단 순서 변경
- 긴 문장 분리
- 중복 표현 삭제
- 명시된 행위자의 주어 복원
- 표준 용어 적용
- 사실보다 강한 동사를 약화
- 원문 근거를 해당 주장 옆으로 이동

### 사용자 확인 또는 근거가 필요한 변경

- 새로운 실행주체·협력기관 지정
- 예산·기간·목표치 추가
- 기능과 자동화 범위 확장
- 원인의 확정
- 비교대안과 우월성 추가
- 법적·기술적 가능성 확정

### 금지할 변경

- 그럴듯한 출처, 인터뷰, 통계를 생성하기
- 계획을 완료 사실로 바꾸기
- 사용자가 명시한 수치나 범위를 조용히 변경하기
- 불리한 제약과 반례를 삭제해 확실성을 높이기
- 인용문의 뜻을 바꾸는 요약을 만들기

## 결함 분류

가장 근본적인 원인 하나로 분류하라.

| 분류 | 사용 조건 | 기본 조치 |
| --- | --- | --- |
| `CONTENT_GAP` | 필요한 주장·계획·비교·지표 자체가 없음 | 조사 질문 또는 의사결정 요청 |
| `WRITING_GAP` | 정보는 있으나 위치·표현·연결이 약함 | 재배치·분리·구체화 |
| `EVIDENCE_RISK` | 출처·수치·인과·주장 상태가 불명확함 | 검증·범위 축소·표시 |
| `COMPLIANCE_RISK` | 공식 조건 위반 또는 미확인 | 공고 원문 확인 후 우선 해결 |

하나의 문장이 여러 문제를 만들면 주된 결함 하나를 지정하고 보조 영향을 적어라. 예를 들어 출처 없는 효과 수치는 `EVIDENCE_RISK`가 주된 결함이며 `어조`는 연관 영향이다.

## 제출조건과 권리 위험

공고에서 다음을 반드시 추출하라.

- 참가 자격과 팀 구성
- 공모 분야와 제외 대상
- 동일·유사 제안, 기수상작, 중복 응모 조건
- 저작권·초상권·데이터 이용권
- 생성형 AI 사용·표시 규정이 있는지
- 개인정보와 비공개 정보 포함 여부
- 파일 형식, 파일명, 병합, 서명, 용량
- 페이지·글자 수와 서식
- 마감 날짜, 시각, 시간대, 제출 경로
- 필수 첨부와 누락 시 처리

불명확한 권리 관계를 `문제없다`고 단정하지 마라. 법률 판단이 필요한 경우 위험과 확인 대상을 분리해 제시하라.

## 최종 무결성 감사

제출 전 다음을 순서대로 확인하라.

1. 공식 공고와 평가표의 최신 판본을 사용했는가
2. 모든 핵심 수치에 단위·시점·대상·출처가 있는가
3. 사실·추정·목표·계획이 동사 수준에서 구분되는가
4. 문제 근거가 특정 해법의 효과 근거로 오용되지 않았는가
5. 인과 주장이 실제 근거보다 강하지 않은가
6. 비교대안이 같은 기준으로 비교되었는가
7. 제목·요약·본문·표의 대상과 수치가 일치하는가
8. 실행주체, 일정, 예산, 협력 상태가 일치하는가
9. 미확정 항목이 자연스러운 문장 속에 숨지 않았는가
10. 한계가 삭제되지 않으면서 주장 뒤에 반복되어 결론을 흐리지 않는가
11. 제출·권리·중복응모 조건을 확인했는가

한 항목이라도 핵심 사실을 무너뜨리면 문체 개선보다 먼저 보고하라.
