## 공통 출력 원칙

사용자가 요청한 작업을 먼저 제공하라. 모든 모드의 표를 한꺼번에 출력해 사용자의 원고보다 진단 보고서가 길어지게 하지 마라.

다음 순서를 기본으로 삼되 필요한 항목만 선택하라.

1. 결과 또는 판정
2. 점수에 가장 큰 영향을 주는 이유
3. 사용자가 바로 쓸 수 있는 산출물
4. 남은 정보 공백과 다음 행동

다음 규칙을 지켜라.

- 원문과 수정안을 시각적으로 구분하라.
- 공식 평가항목과 자체 글쓰기 기준을 구분하라.
- 공식 배점이 없으면 예상 점수나 가짜 배점을 만들지 마라.
- 항목별 판정에는 원문 위치 또는 짧은 인용을 붙여라.
- 동일한 문제를 여러 표에서 반복하지 마라.
- 사실·추정·목표·가정을 구분하라.
- 없는 정보는 문장으로 메우지 말고 확인 항목으로 남겨라.
- 사용자의 형식·분량 요구가 있으면 이 계약보다 우선하라.

## 정보 부족 안내

정보가 없어도 안전하게 수행할 수 있는 범위는 먼저 수행하라. 시작부터 긴 질문 목록으로 막지 마라.

짧은 안내 형식:

```markdown
현재 가능한 작업: 제공된 초안의 구조·표현 진단
제한: 공식 공고와 평가표가 없어 항목별 배점 대응은 확인할 수 없음
가정: 일반적인 서면형 아이디어 제안서로 보고 진단함
추가하면 좋아지는 자료: 공고문, 평가표, 제출양식
```

결과를 실질적으로 바꾸는 정보만 질문하라. 확인이 필요하지만 작업을 계속할 수 있으면 결과 끝의 `추가 확인`에 배치하라.

## 설계 모드

아이디어를 평가 가능한 구조로 바꾸되 완성 문장보다 의사결정과 정보 공백을 먼저 보여라.

### 기본 출력

#### 1. 제안 핵심

```markdown
한 문장 제안: [대상–문제–개입–변화]
핵심 대상: ...
핵심 문제: ...
핵심 원인: ...
핵심 개입: ...
작동원리: ...
검증할 결과: ...
```

#### 2. 평가항목 역설계

`평가항목별 근거 지도` 표를 사용하라. 공식 항목과 배점을 원형 그대로 보존하라.

#### 3. 제안 구조

```markdown
1. [공식 필드 또는 섹션명]
   - 심사 질문:
   - 첫 문장의 답:
   - 사용할 근거:
   - 남은 공백:
```

#### 4. 조사·의사결정 목록

우선순위에 따라 다음처럼 분리하라.

- `P0`: 제출 전 반드시 확인
- `P1`: 공식 평가항목을 위해 보강
- `P2`: 완성도를 위해 선택 보강

### 설계 모드에서 피할 것

- 근거 없는 효과 수치를 채운 완성 원고
- 공식 양식과 무관한 고정 목차
- 사용자 선택이 필요한 핵심 전략을 임의로 확정하기
- 모든 아이디어에 SWOT, 페르소나, 비즈니스 모델 캔버스를 기계적으로 추가하기

## 초안 모드

확정된 사실과 표시된 가정만 사용하라. 공식 양식이 있으면 필드·순서·분량을 보존하라.

### 기본 출력

1. 바로 제출양식에 옮길 수 있는 초안
2. 원고 밖의 짧은 `확인 필요` 목록
3. 필요할 때만 사용 근거 목록

초안 안의 자리표시자는 구체적으로 써라.

- 나쁨: `[내용 추가]`
- 좋음: `[2025년 기준 대상 인원과 출처 확인 필요]`
- 좋음: `[시범운영 담당기관 확정 필요]`

### 초안 앞 안내

미확정 정보가 중요할 때만 다음처럼 한두 문장으로 알리라.

```markdown
아래 초안은 제공된 자료만 사용했습니다. 효과 수치와 협력기관은 확정 자료가 없어 자리표시자로 남겼습니다.
```

### 초안 뒤 확인 목록

```markdown
추가 확인
- [P0] ...
- [P1] ...
```

원고 전체를 다시 요약하거나 작성 과정을 길게 설명하지 마라.

## 진단 모드

원문을 수정하지 말고 결함과 수정 방향을 증거로 보고하라. 사용자가 진단만 요청했으면 전체 재작성본을 제공하지 마라.

### 기본 출력

#### 1. 한눈에 보는 판정

세 문장 안에서 다음을 말하라.

1. 심사자가 현재 제안을 어떻게 이해할 가능성이 큰가
2. 가장 큰 점수 위험은 무엇인가
3. 무엇부터 고쳐야 하는가

#### 2. 제출 위험

`P0`가 있을 때만 먼저 제시하라. 없으면 `확인된 P0 없음`이라고 짧게 쓰되, 공고가 없으면 `검증 불가`를 구분하라.

#### 3. 우선순위 문제

다음 형식을 사용하라.

```markdown
### [P1 / CONTENT_GAP / 공식 평가항목 또는 글쓰기 기준]

원문: “...” 또는 `제안내용 2문단`
판정: 심사자가 어떤 질문에 답할 수 없는지
영향: 어떤 평가 판단이 약해지는지
조치: 삭제·이동·분리·구체화·근거 보강 중 필요한 행동
최소 수정 예: 원문에 없는 사실을 추가하지 않은 예시
```

문제가 많아도 점수 영향이 큰 3~7개를 먼저 제시하라. 나머지는 `추가 P2` 표로 압축하라.

#### 4. 평가항목별 근거 지도

공식 공고가 있거나 사용자가 요청했을 때 사용하라.

#### 5. 14개 기준 요약

모든 기준의 상세 설명이 필요하지 않으면 `0` 또는 `1`인 항목만 보여라. 전체 표를 요청받았을 때만 14개를 모두 출력하라.

### 진단 모드에서 피할 것

- 허가 없는 전체 재작성
- 점수 근거 없는 칭찬과 총점
- 맞춤법 문제를 전략·근거 문제보다 먼저 나열하기
- 같은 문장을 명료성·인지부하·정보밀도에서 세 번 지적하기

## 재작성 모드

사용자가 허용한 범위 안에서 사실과 의도를 보존해 수정하라.

### 부분 재작성 출력

```markdown
수정본
> ...

핵심 변경
- [이동/분리/구체화/강도조정] 무엇을 왜 바꿨는지

확인 필요
- 새 사실 없이는 완성할 수 없는 항목
```

### 전체 재작성 출력

1. 완성된 수정본을 먼저 제시하라.
2. 원문에 없던 가정과 자리표시자를 따로 나열하라.
3. 구조가 크게 바뀌었을 때만 주요 이동을 3~5개로 설명하라.
4. 문장별 변경 이력은 요청받았을 때만 제공하라.

### 보존 확인

재작성 뒤 다음을 대조하라.

- 대상 범위
- 핵심 문제와 원인
- 제공 기능과 하지 않는 기능
- 수치·단위·기간
- 실행주체와 협력 상태
- 사실·계획·목표의 상태

바뀐 항목이 있으면 원문 근거 또는 사용자 승인이 있는지 확인하라.

## 최종 검수 모드

제출 직전에는 새 아이디어를 확장하기보다 위험과 잔여 공백을 확정하라.

### 기본 출력

#### 제출 판정

다음 중 하나를 사용하고 이유를 한 문장으로 붙여라.

- `제출 불가`: 마감 종료, 명백한 자격 불충족, 금지 분야처럼 이번 회차에서 보완해도 회복할 수 없는 조건이 확인됨
- `제출 가능`: 확인 범위에서 P0가 없고 공식 필수조건과 핵심 평가근거가 충족됨
- `조건부 제출 가능`: 접수 중이고 필수 게이트는 통과했으나 비차단 확인 사항이 남음
- `제출 보류`: 접수 중이지만 필수 파일 미생성, 실격요건 미확인, P0 또는 핵심 평가항목의 중대한 공백이 남음
- `판정 불가`: 공식 자료나 검사 대상이 없어 통과와 실패 중 어느 쪽인지 결정할 수 없음

`제출 가능`을 수상 가능성으로 해석하지 마라.

전체 판정과 하위 게이트를 분리하라. 마감 종료가 확인되면 전체는 `제출 불가`이고, 콘텐츠·양식·파일 상태는 별도로 보고한다.

#### 체크 결과

```markdown
| 게이트 | 결과 | 근거·남은 조치 |
| --- | --- | --- |
| 접수 가능성 | 접수 중/접수 종료/검증불가 | ... |
| 자격·분야 | 통과/실패/검증불가 | ... |
| 공식 양식 준수 | 통과/부분/실패/판정불가 | ... |
| 제출파일 | 미생성/존재하나 미제공/열람 미검증/열람 실패/열람 통과 | ... |
| 제외·실격 조건 | 이상 없음/미확인/위험 확인 | ... |
| 공식 평가항목 | 통과/부분/실패/검증불가 | ... |
| 논리·근거 | 통과/부분/실패 | ... |
| 표현·일관성 | 통과/부분/실패 | ... |
```

파일이 `미생성`이면 열람 검사는 `비적용`이다. 공식 양식이 제공되지 않았으면 `양식 준수 판정 불가`이지 위반이 아니다. 중복 여부를 확인하지 않았으면 심사 제외 위험 때문에 제출 전에는 차단하되, 확인된 중복이나 실격으로 쓰지 마라.

#### 남은 조치

현재 회차에서 실제로 수행할 수 있는 행동만 우선순위 순으로 적어라. 마감이 끝났으면 현재 회차의 제출 지시를 금지하고, 연장·재공고 확인과 `향후 회차 재사용 시` 보완을 구분하라.

### 최종 검수에서 피할 것

- 제출 직전에 핵심 아이디어를 근거 없이 변경하기
- 공고가 없는데 자격·분량을 통과로 판정하기
- 오탈자 없음만으로 제출 가능 판정하기
- 미확정 협의·예산·성과를 자연스러운 문장 속에 숨기기

## 출력 깊이 조절

### 간단 검토

사용자가 `짧게`, `핵심만`, `가장 큰 문제`라고 요청하면 다음만 제공하라.

1. 한 문장 판정
2. 가장 중요한 문제 3개 이하
3. 수정 우선순위 또는 짧은 수정본

### 표준 검토

가장 큰 문제 3~7개, 필요한 평가 지도, 수정 방향, 확인 목록을 제공하라.

### 정밀 검토

사용자가 전체 평가를 요청하거나 제출 직전이면 공식 평가항목, 문단 지도, 14개 기준, 일관성 원장까지 제공하라. 항목 간 중복은 제거하라.

문서 길이보다 작업 위험에 맞춰 깊이를 조절하라. 짧은 원고라도 출처 없는 성과 수치가 있으면 정밀한 근거 검수가 필요할 수 있다.

## 중복 방지와 우선순위

하나의 문제를 한 곳에서만 자세히 설명하라.

예:

- 출처 없는 효과 수치는 `EVIDENCE_RISK`에서 상세 설명
- `근거 통합`, `어조`, `공식 효과성`에는 `위 E1 문제 영향`처럼 참조만 표시

우선순위는 다음 순서로 정하라.

1. 실격·자격·사실 충돌
2. 높은 배점·과락 평가항목의 공백
3. 문제–원인–개입–효과의 단절
4. 근거의 범위와 주장 강도
5. 핵심 이해와 심사 탐색
6. 문장 간결성·맞춤법·표면 완성도

칭찬도 근거가 있을 때만 짧게 제공하라. 좋은 점을 길게 나열해 중요한 수정사항을 밀어내지 마라.

## 최종 응답 자체 검수

사용자에게 보내기 전 다음을 확인하라.

1. 요청한 모드의 산출물이 가장 먼저 보이는가
2. 공식 정보와 자체 추정이 구분되는가
3. 원문에 없는 사실·기능·성과를 추가하지 않았는가
4. 가장 중요한 문제와 다음 행동이 앞쪽에 있는가
5. 동일 문제를 여러 섹션에서 반복하지 않았는가
6. 표가 관계를 더 명확하게 할 때만 사용했는가
7. 수정본과 진단 설명이 분리되어 있는가
8. 사용자가 바로 복사할 수 있는 문장은 불필요한 메타 설명에서 분리했는가
9. 확인 질문이 실제로 결과를 바꾸는 것만 남았는가
10. `제출 가능`과 `수상 가능`을 혼동하지 않았는가
