## 공통 원칙

사용자가 바로 사용할 결과를 먼저 제공하라. 모든 완성 원고와 구조 설계는 다음 7부 구조만 사용하라.

`제안 요약 → 문제 상황 → 문제의 원인 → 해결 방안 → 차별성과 실현 가능성 → 기대 효과 → 결론`

모든 문단은 `주장 → 설명 → 근거·사례 → 소결론`만 사용하라.

다음을 모든 모드에 적용하라.

- 공식 평가항목과 자체 보조 질문을 구분하라.
- 원문과 수정본을 명확히 구분하라.
- 공식 양식이 있으면 필드명과 순서를 유지하고 일곱 기능을 대응시켜라.
- 없는 사실을 만들지 마라.
- 공식 배점이 없으면 예상 점수나 수상 확률을 만들지 마라.
- 웹 자료를 사용하면 핵심 근거 가까이에 원문 출처를 붙여라.

## 2. 초안 모드

첫 산출물은 제출양식에 옮길 수 있는 완성 원고다.

### 출력 순서

1. 제목
2. 제안 요약
3. 문제 상황
4. 문제의 원인
5. 해결 방안
6. 차별성과 실현 가능성
7. 기대 효과
8. 결론
9. 원고에 실제 사용한 출처가 있을 때만 출처

필수 입력값이 없어 원고가 성립하지 않을 때만 정확한 위치에 자리표시자를 사용하라. 초안 뒤에 일반적인 작성 과정이나 중복 진단을 붙이지 마라.

## 3. 진단 모드

첫 산출물은 현재 원고의 가장 치명적인 구조·논리 결함이다.

### 한눈에 보는 판정

세 문장으로 제시하라.

1. 현재 원고가 주장하는 핵심 결론
2. 독자의 이해 또는 공식 평가를 가장 크게 막는 결함
3. 가장 먼저 바꿀 구조나 문단

| --- | --- | --- | --- | --- | --- |
| P0/P1/P2 |  |  |  |  |  |

### 즉시 교체 예시

원문에 없는 사실을 추가하지 않고 `주장 → 설명 → 근거·사례 → 소결론`으로 교체한 문단을 제시하라.

문제가 많아도 영향이 큰 3~7개만 먼저 다루고 같은 문제를 여러 기준에서 반복하지 마라.

## 4. 재작성 모드

첫 산출물은 7부 구조의 완성 수정본이다.

### 출력 순서

1. 제목과 완성 수정본
2. 구조가 크게 바뀐 경우에만 핵심 변경 3~5개
3. 원문에 없던 필수 정보가 필요한 경우에만 확인 항목

재작성 전에 내부적으로 핵심 결론, 문제–원인–해결 대응, 7부 구조를 확정하라. 사용자가 설계표도 요청했을 때만 수정본 뒤에 압축표를 제공하라.

### 핵심 변경 태그

- `[핵심 결론 명확화]`
- `[문제–원인 구분]`
- `[원인–해결 연결]`
- `[해결 방안 구체화]`
- `[차별성 비교]`
- `[실현 가능성 보강]`
- `[효과 측정]`
- `[문단 공식 통일]`
- `[문장 명료화]`

## 5. 최종 검수 모드

첫 산출물은 제출을 막거나 약화하는 잔여 결함과 즉시 교체문이다.

### 최종 판정

- `제출 준비 완료`: 공식 조건, 7부 구조, 문단 공식, 근거와 측정이 모두 닫힘
- `즉시 보강 후 제출`: 현재 자료로 교체 가능한 핵심 결함이 남음
- `회차 제출 불가`: 마감 종료 또는 명백한 자격 불충족이 확인됨
- `공식자료 확인 필요`: 공고·양식이 없어 제출 조건을 판정할 수 없음

## 7. 출력 깊이

### 짧게

한 문장 판정, 가장 큰 문제 3개 이하, 즉시 교체문만 제공하라.

### 표준

요청 산출물, 문제–원인–해결의 핵심 연결, 공식 평가항목 지도, 중요한 보강 3~7개를 제공하라.

### 정밀

공식 평가항목 전체, 7부 구조 전체, 실행 설계, 차별성 비교, 기대 효과 측정, 전체 검수표를 적용하라.

## 8. 최종 응답 검사

- 사용자가 요청한 결과가 첫 번째인가
- 완성 원고가 유일한 7부 구조를 따르는가
- 모든 문단이 단일 문단 공식을 따르는가
- 문제와 원인이 구분되는가
- 해결 방안이 원인과 직접 연결되는가
- 차별성과 실현 가능성이 구체적인가
- 직접·확장·장기 효과와 측정 기준이 있는가
- 공식 평가항목에 필요한 판단과 근거가 있는가
- 문장이 짧고 구체적이며 용어가 일관되는가
- 결론에 새로운 내용이 없는가
- 원문에 없는 사실을 만들지 않았는가
