# 공모전 작업 모드별 출력 계약

## 공통 원칙

사용자가 바로 쓰는 결과를 먼저 제공하라. 공모전 원고보다 작업 설명과 진단표가 길어지게 하지 마라.

기본 순서:

1. 요청한 산출물
2. 확정 논리사슬 또는 가장 큰 논리 단절
3. 평가항목별 득점 연결
4. 즉시 실행할 보강 행동

다음은 모든 모드에 적용한다.

- 공식 평가항목과 자체 판단축을 분리한다.
- 원문과 수정본을 명확히 구분한다.
- 핵심 주장은 `주장 → 근거 → 논거 → 판정`으로 완결한다.
- 사용자에게 제출할 문장은 단정형·능동형·결론 우선으로 쓴다.
- 자기반박, 관성적 유보, 장황한 조건 설명을 기본 출력에 넣지 않는다.
- 없는 사실을 만들지 않는다.
- 공식 배점이 없으면 가짜 점수나 수상 확률을 만들지 않는다.
- 웹 조사 결과를 사용하면 핵심 주장 가까이에 원문 출처를 붙인다.

## 1. 설계 모드

첫 산출물은 `확정 논리사슬`이다.

### 출력 순서

#### 채택명제

> 심사위원회는 `[대상]`의 `[문제]`를 `[메커니즘]`으로 바꾸어 `[성과]`를 만드는 본 제안을 선택해야 한다.

#### 확정 논리사슬

| 고리 | 확정 명제 | 근거 | 논거 | 다음 연결 | 공식 평가항목 |
| --- | --- | --- | --- | --- | --- |
| 채택 결론/목적/문제/원인/기존 공백/개입/메커니즘/산출/성과/비교우위/실행/득점 결론 |  |  |  |  |  |

#### 평가항목 득점 지도

| 공식 항목 | 배점 | 최고점 판정 문장 | 핵심 증거 | 원고 위치 |
| --- | ---: | --- | --- | --- |
|  |  |  |  |  |

#### 원고 구조

| 섹션·공식 필드 | 첫 판정 문장 | 사용할 논리사슬 고리 | 핵심 근거 | 닫는 심사 결론 |
| --- | --- | --- | --- | --- |
|  |  |  |  |  |

필요한 자료가 있으면 논리사슬을 먼저 가능한 범위까지 완성한 뒤 `즉시 확보할 증거`만 우선순위로 제시하라.

## 2. 초안 모드

첫 산출물은 바로 제출양식에 옮길 수 있는 완성 원고다.

### 출력 순서

1. 제목
2. 제출용 원고
3. 사용한 핵심 출처가 있을 때만 출처
4. 필수 입력값 하나가 비어 원고가 성립하지 않을 때만 정확한 자리표시자

원고를 다음 흐름으로 쓴다.

`채택 결론 → 문제·시급성 → 원인·기존 공백 → 개입·메커니즘 → 비교우위 → 실행 → 성과·측정 → 최종 채택 결론`

초안 뒤에 일반적인 주의사항이나 작성 과정 설명을 붙이지 마라.

## 3. 진단 모드

첫 산출물은 `현재 원고가 왜 선택되지 않는지`에 대한 가장 치명적인 결론이다.

### 출력 순서

#### 한눈에 보는 판정

세 문장으로 제시한다.

1. 현재 원고가 주장하는 채택명제
2. 그 명제를 무너뜨리는 가장 큰 논리 단절 또는 득점 손실
3. 가장 먼저 바꿀 구조

#### 논리사슬 진단

| 우선순위 | 끊긴 고리 | 원문 위치 | 심사자가 못 내리는 판단 | 교체 명제·필요 근거 |
| --- | --- | --- | --- | --- |
| P0/P1/P2 |  |  |  |  |

#### 즉시 교체 예시

원문에 없는 사실을 추가하지 않고 판정문·논거·연결을 강화한 예시를 제시하라.

문제가 많아도 점수 영향이 큰 3~7개만 먼저 다룬다. 같은 문제를 여러 기준에서 반복하지 마라.

## 4. 재작성 모드

첫 산출물은 완성 수정본이다.

### 출력 순서

1. 수정본
2. 구조가 크게 바뀐 경우에만 핵심 변경 3~5개
3. 원문에 없던 필수 정보가 필요한 경우에만 정확한 확인 항목

재작성 전에 내부적으로 채택명제와 논리사슬을 확정한다. 사용자가 논리사슬도 요청했으면 수정본 다음에 압축표로 보여준다.

### 핵심 변경 태그

- `[결론 선점]`
- `[원인–개입 연결]`
- `[논거 보강]`
- `[비교우위 확정]`
- `[실행 증명]`
- `[성과 측정]`
- `[평가항목 회수]`
- `[완곡어 제거]`

## 5. 최종 검수 모드

첫 산출물은 `제출문을 약화하는 잔여 결함`과 즉시 교체문이다.

### 최종 판정

- `제출 준비 완료`: 공식 필수조건을 충족하고 논리사슬과 평가항목별 근거가 닫힘
- `즉시 보강 후 제출`: 교체 가능한 핵심 고리나 필수 입력값이 남음
- `회차 제출 불가`: 마감 종료 또는 명백한 자격 불충족이 확인됨
- `공식자료 확인 필요`: 공고·양식이 없어 제출 조건 자체를 판정할 수 없음

### 압승 감사표

| 게이트 | 판정 | 결정적 근거 | 즉시 조치 |
| --- | --- | --- | --- |
| 채택명제 | 장악/약함/붕괴 |  |  |
| 논리사슬 | 장악/약함/붕괴 |  |  |
| 비교우위 | 장악/약함/붕괴 |  |  |
| 실행 증명 | 장악/약함/붕괴 |  |  |
| 성과 측정 | 장악/약함/붕괴 |  |  |
| 평가항목 회수 | 장악/약함/붕괴 |  |  |
| 문장 강도 | 장악/약함/붕괴 |  |  |
| 공식 제출조건 | 통과/실패/확인 필요 |  |  |

`즉시 보강 후 제출`이면 현재 회차에서 바로 할 수 있는 행동만 순서대로 적어라.

## 7. 출력 깊이

### 짧게

한 문장 판정, 가장 큰 문제 3개 이하, 즉시 교체문만 제공한다.

### 표준

요청 산출물, 논리사슬 핵심 고리, 평가항목 지도, 가장 중요한 보강 3~7개를 제공한다.

### 정밀

공식 평가항목 전체, 논리사슬 전체, 비교우위, 실행 증명, 성과 측정, 15개 루브릭을 적용한다.

## 8. 최종 응답 자체 감사

- 사용자가 요청한 산출물이 첫 번째인가
- 채택명제가 한 문장으로 보이는가
- 논리사슬이 정방향·역방향으로 닫히는가
- 모든 핵심 주장에 근거와 논거가 있는가
- 비교우위가 같은 기준에서 증명되는가
- 실행과 측정이 구체적인가
- 모든 공식 평가항목에 판정 문장이 있는가
- 문장이 단정형·능동형·결론 우선인가
- 자기반박과 관성적 유보가 제거됐는가
- 원문에 없는 사실을 만들지 않았는가
